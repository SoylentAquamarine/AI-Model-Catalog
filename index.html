AI Model Catalog

A public, normalized JSON catalog of AI providers and models: pricing, free/paid status, capabilities with evidence, context limits — and exactly how to connect to each provider. Plain HTTPS. No account, no API key, no rate-limit games.

The situation this solves

Every AI provider publishes model information differently. Some have clean model APIs, some publish pricing only on web pages, some expose almost nothing machine-readable. Anyone building an app that talks to more than one provider ends up re-scraping and re-guessing the same facts: which models exist, what they cost, which are free, what they can do, and how to call them.

This project does that work once, in public. GitHub Actions checks the providers (weekly, on demand, and when providers email announcements), normalizes what they expose into one schema, validates it, and publishes plain JSON files anyone can download. Two integrity rules govern the data:

  • Never present guessed data as confirmed. Every capability claim carries its evidence: value, source, confidence, lastChecked. Values can be true, false, "likely", "partial", or an honest "unknown".
  • No user data, ever. The catalog describes providers. Your API keys, account state, and preferences stay on your machine.

Quick start

# 1. What exists, and when was the catalog last built?
curl -s https://raw.githubusercontent.com/SoylentAquamarine/AI-Model-Catalog/main/catalog-index.json

# 2. Everything about one provider: models, pricing, capabilities, how to connect
curl -s https://raw.githubusercontent.com/SoylentAquamarine/AI-Model-Catalog/main/providers/openrouter.json

That is the entire consumption surface. The scripts, workflows, and docs in the repo are maintenance machinery — consumers never need them.

Using the files in your app

Each provider file answers two questions: how do I talk to this provider, and what does it offer.

1. How to connect: apiType + auth

Implement one client driver per apiType and you can talk to every provider in the catalog with zero provider-specific code:

openai-compatible  POST {apiBase}/chat/completions
anthropic          POST {apiBase}/messages
gemini             POST {apiBase}/models/{model}:generateContent
ollama             POST {apiBase}/api/chat  (local runtime)

The auth block says how to attach your key (the catalog never contains keys):

"auth": { "scheme": "bearer" }          → Authorization: Bearer <key>
"auth": { "scheme": "header",
          "name": "x-api-key",
          "extraHeaders": {...} }        → x-api-key: <key> + extras verbatim
"auth": { "scheme": "query",
          "name": "key" }                → ?key=<key>
"auth": { "scheme": "none" }            → no credential needed

2. What it offers: models[]

Route work using the model entries:

costClass    free | paid | trial | local | unknown
pricing      USD per million tokens, when known
limits       contextTokens / maxOutputTokens
status       available | preview | deprecated
capabilities chat, vision, tools, json,
             reasoning, embeddings, ... each with
             { value, source, confidence, lastChecked }

Treat capability values honestly: "likely" and "unknown" are not true. A free-model router, for example, filters costClass == "free" and capabilities.chat.value === true.

3. Merge with your own local state

The catalog knows what exists publicly. It cannot know what your key can access, your rate-limit state, or which providers you have enabled. Keep those in your own config files, keyed by provider id, and merge at load time. Pattern that works well: store downloaded catalog files read-only (overwrite them wholesale on refresh) and keep a small local per-provider file next to them holding your key reference, an enabled flag, and a default model — refreshes can then never destroy local settings.

Building an updater into your app

Syncing is four steps — fetch the index, compare, download, swap:

const BASE = "https://raw.githubusercontent.com/SoylentAquamarine/AI-Model-Catalog/main/";

async function syncCatalog(store) {                     // store = your app's persistence
  const index = await (await fetch(BASE + "catalog-index.json")).json();

  if (index.generatedAt === store.lastSync) return;     // nothing new: one ~1 KB request, done

  const major = parseInt(index.schemaVersion, 10);
  if (major > 1) {                                      // newer breaking schema than this code knows
    console.warn("catalog schema " + index.schemaVersion + " is newer; keeping cached copy");
    return;                                             // degrade gracefully, never crash
  }

  for (const p of index.providers) {
    const file = await (await fetch(BASE + p.file)).json();
    await store.writeTemp(p.id, file);                  // download everything first...
  }
  await store.swapTempIntoPlace();                      // ...then swap atomically: never half-updated
  store.lastSync = index.generatedAt;
}

The same flow is a dozen lines in Python or Go. Rules that keep you unbreakable:

  • Ignore fields you don't recognize. New fields arrive as minor versions; well-behaved consumers never notice.
  • Check the major version. On a newer major, keep your cached copy and warn — don't fail hard.
  • Fail safe. No internet or GitHub down? Your app keeps running on the files it already has.

Cadence: the upstream catalog regenerates weekly, plus on-demand runs and provider announcement emails. Checking on app startup, or daily, is plenty — the index fetch costs ~1 KB, so even hourly checks are harmless. Pin to a release tag instead of main if you need reproducible data.
