What this is, in one sentence

It's a big, tidy phone book of the free AI providers. For every one — Groq, Google Gemini, Mistral, Cerebras, OpenRouter, and a dozen more — it lists their models, which are free, what each one can do, and the exact address and rules for calling it. The whole thing is kept up to date automatically and published as plain config files anyone can download — no account, no API key, no sign-up. The goal is simple: make the scattered world of free AI easy to use, in one standard format.

How we know what's actually free: this project pays for nothing. Every provider is checked with a free-tier or keyless account only — never a paid plan — so the catalog is a live map of what you can genuinely reach and run for free right now. Paid models that show up are still listed and clearly flagged, so you see the whole picture — but the mission is the free tier.

The problem it solves

Say you're building something that uses AI — a chatbot, a tool, an app with a "pick a model" dropdown. The moment you want to support more than one provider, you hit the same wall everyone hits: every provider describes its models differently.

One publishes a clean list of models. Another puts its prices on a marketing web page you'd have to scrape. A third barely tells you anything a computer can read. So every developer ends up doing the same tedious detective work over and over — hunting down which models exist, what they cost, which are free, what they can do, and how to actually call them.

And if you're trying to build on AI without a budget, there's an extra headache: which providers even have a free tier, which models on it are actually usable, and how do you connect to each one? This project answers exactly that — and hands you standardized, always-current config so you can delete that whole chunk of setup code.

Without this catalog

  • Visit 15 different provider websites
  • Copy prices and model names by hand
  • Guess at what each model can do
  • Re-check it all every few weeks as things change
  • Repeat for every project you build

With this catalog

  • Download one small file to see every provider
  • Download one file per provider for the details
  • Prices, free/paid, and capabilities already sorted
  • Kept current for you, automatically
  • Reuse it across every project — it's just data

This project does that detective work once, in public. A robot (GitHub Actions) checks the providers on a schedule, tidies everything into one consistent format, double-checks it, and publishes the result as plain files. You just download them.

Two promises about the data

1. It never fakes certainty. If we know a model supports something because the provider says so, it's marked confident. If it's an educated guess, it's marked as a guess. Every capability comes with its evidence — where the fact came from and how sure we are — so you can trust it or verify it, but you're never misled.

2. It never touches your private stuff. The catalog only describes providers that are public. Your API keys, your account, your usage — all of that stays on your own machine. There's nothing personal in here.

How to use it

There are three levels, from "just look" to "build it into your app." Pick the one that fits.

  1. Just browse it

    You're already doing it — the table above is the catalog. Click any view link to read a provider's full file in your browser, or raw to grab the plain data.

  2. Download it from the command line

    Two files are all you ever need. First, the index — it lists which providers exist and when the catalog was last built:

    curl -s https://raw.githubusercontent.com/SoylentAquamarine/AI-Model-Catalog/main/catalog-index.json

    Then a single provider's file — everything about its models, prices, capabilities, and how to connect:

    curl -s https://raw.githubusercontent.com/SoylentAquamarine/AI-Model-Catalog/main/providers/openrouter.json

    That's the whole thing. Everything else in the project is behind-the-scenes machinery you never have to touch.

  3. Build it into your app

    Each provider file answers two questions: how do I connect to this provider, and what does it offer? The details are just below.

Bring your own key

Be clear on one thing: the catalog is free to download, but it doesn't come with API keys — and most providers require one. The "no account, no key" promise is about getting the catalog itself. To actually call a provider, you sign up with that provider (almost always free), generate your own API key, and attach it using the auth shape the catalog hands you. The catalog never contains keys and never will — that's your private data and it stays on your machine.

Two exceptions worth knowing:

  • Keyless providers — a few (like Pollinations) use "auth": { "scheme": "none" } and need no key at all.
  • Local models — a local Ollama runs on your own machine with no key (see the Ollama section below).

For everything else, the code is the same shape for every provider — that's the whole point of standardizing it:

  1. Read the provider's apiType and auth from its catalog file.
  2. Look up your key for that provider from your own secure storage (never the catalog).
  3. Attach it the way auth.scheme says — bearer header, custom header, or query parameter.
  4. Send the request to the endpoint for that apiType.
Storing your keys safely on Windows (DPAPI)

Never hard-code an API key or commit it to source control. On Windows the simplest secure-at-rest option is DPAPI (the Data Protection API). It encrypts data so only the current Windows user (or machine) can decrypt it — you manage no encryption keys of your own, and a stolen file is useless on another machine.

C# — System.Security.Cryptography.ProtectedData:

using System.Security.Cryptography;
using System.Text;

// Encrypt once (e.g. when the user pastes their key), then save the bytes to disk:
byte[] Protect(string apiKey) =>
    ProtectedData.Protect(
        Encoding.UTF8.GetBytes(apiKey),
        optionalEntropy: null,
        scope: DataProtectionScope.CurrentUser);   // only this Windows user can read it back

// Decrypt in memory, right before you make the call:
string Unprotect(byte[] blob) =>
    Encoding.UTF8.GetString(
        ProtectedData.Unprotect(blob, null, DataProtectionScope.CurrentUser));

Store the encrypted blob next to your local provider settings (keyed by provider id), and only ever decrypt it into memory at the moment you send a request.

PowerShell — same DPAPI underneath:

# Encrypt and save
Read-Host "Groq API key" -AsSecureString |
  ConvertFrom-SecureString | Set-Content "$env:APPDATA\MyApp\groq.key"

# Load and use
$secure = Get-Content "$env:APPDATA\MyApp\groq.key" | ConvertTo-SecureString
$key = [Runtime.InteropServices.Marshal]::PtrToStringAuto(
  [Runtime.InteropServices.Marshal]::SecureStringToBSTR($secure))

DPAPI ties the encrypted data to the current Windows user and machine, so the file can't be decrypted if it's copied elsewhere — exactly what you want for a personal key. Building cross-platform? Use the OS keychain instead (macOS Keychain, libsecret on Linux) or a dedicated secrets manager.

Reading a provider file

The two parts of every file:

How to connect

apiType tells you which style of API it speaks, and auth tells you how to attach your key. Write one small piece of code per apiType and you can talk to every provider that uses it — no per-provider special-casing.

What it offers

The models list gives each model's cost (costClass: free / paid / trial / local), price when known, size limits, and capabilities like chat, vision, tools, and JSON — each tagged with how confident we are.

The exact connection rules (for developers)

Three API styles cover every provider in the catalog (a fourth, ollama, is supported for local models — see below):

openai-compatible  POST {apiBase}/chat/completions
anthropic          POST {apiBase}/messages
gemini             POST {apiBase}/models/{model}:generateContent

The auth block says how to attach your key. The catalog never contains keys — only the shape:

"auth": { "scheme": "bearer" }          → Authorization: Bearer <key>
"auth": { "scheme": "header",
          "name": "x-api-key",
          "extraHeaders": {...} }        → x-api-key: <key> + extras verbatim
"auth": { "scheme": "query",
          "name": "key" }                → ?key=<key>
"auth": { "scheme": "none" }            → no credential needed

And the fields on each model entry:

costClass    free | paid | trial | local | unknown
pricing      USD per million tokens, when known
limits       contextTokens / maxOutputTokens
status       available | preview | deprecated
capabilities chat, vision, tools, json, reasoning, embeddings, …
             each with { value, source, confidence, lastChecked }

Treat the capability values honestly: "likely" and "unknown" are not the same as true. A free-model router, for instance, would keep only models where costClass == "free" and capabilities.chat.value === true.

Using a local model with Ollama (not in the catalog)

Ollama runs AI models on your own computer. It's deliberately not in the catalog table above, and never appears in the model tests — because which models it has depends entirely on what you've pulled onto your machine, so a public list can't know and can't verify it. What the catalog does keep is the ability to connect to it: ollama is a supported apiType, so you can wire your local Ollama in right alongside the hosted providers.

You describe it yourself, in your own local config, using the same shape as a catalog provider (this entry never comes from the catalog — you add it):

{
  "id": "ollama",
  "displayName": "Ollama (local)",
  "apiBase": "http://localhost:11434",
  "apiType": "ollama",
  "auth": { "scheme": "none" }
}

Then ask your running Ollama what models it currently has, and chat with one — models change as you pull or remove them, so always read the list at runtime rather than hard-coding it:

GET   http://localhost:11434/api/tags     → the models installed right now
POST  http://localhost:11434/api/chat     → chat with one (the "ollama" driver)

Keep this entry in your own local config next to the downloaded catalog files (see the next section), and your app can treat it as just one more provider — a catalog refresh will never touch it, and it keeps working with no internet at all.

Keeping your own settings separate from the catalog

The catalog knows what exists publicly. It can't know what your key can access, your rate-limit state, or which providers you've turned on. Keep those in your own config, keyed by each provider's id, and combine them when your app loads. A pattern that works well: store the downloaded catalog files as read-only (overwrite them wholesale on each refresh) and keep a small file of your own settings next to them — your key reference, an on/off flag, a default model. That way a refresh can never wipe out your settings.

Auto-syncing the catalog inside your app

Staying up to date is four steps — fetch the index, check if it changed, download, swap:

const BASE = "https://raw.githubusercontent.com/SoylentAquamarine/AI-Model-Catalog/main/";

async function syncCatalog(store) {                     // store = your app's persistence
  const index = await (await fetch(BASE + "catalog-index.json")).json();

  if (index.generatedAt === store.lastSync) return;     // nothing new: one ~1 KB request, done

  const major = parseInt(index.schemaVersion, 10);
  if (major > 1) {                                      // newer breaking format than this code knows
    console.warn("catalog schema " + index.schemaVersion + " is newer; keeping cached copy");
    return;                                             // degrade gracefully, never crash
  }

  for (const p of index.providers) {
    const file = await (await fetch(BASE + p.file)).json();
    await store.writeTemp(p.id, file);                  // download everything first...
  }
  await store.swapTempIntoPlace();                      // ...then swap in one go: never half-updated
  store.lastSync = index.generatedAt;
}

Three rules keep your app unbreakable:

  • Ignore fields you don't recognize. New fields arrive in minor updates; well-behaved apps never notice.
  • Check the major version. If it jumps to a newer major than your code knows, keep your cached copy and warn — don't fail hard.
  • Fail safe. No internet, or GitHub down? Your app keeps running on the files it already has.

How often to check: the catalog regenerates weekly, plus on-demand and when providers announce changes. Checking on app startup, or once a day, is plenty — the index is only ~1 KB, so even hourly is harmless. Want reproducible data? Pin to a release tag instead of main.
